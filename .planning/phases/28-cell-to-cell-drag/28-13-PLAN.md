<objective>
Fix the 'no accent outline on hovered drop cell' portion of Gap 3 (28-HUMAN-UAT Test 4).

**Root cause (from `.planning/debug/zone-visuals-broken.md`):** `LeafNode.tsx:577-583` applies `ringClass = 'ring-2 ring-[#3b82f6] ring-inset'` to the ROOT div via className. Tailwind's `ring-inset` compiles to `box-shadow: inset 0 0 0 2px color`. The root div contains a child `<div className="absolute inset-0 overflow-hidden">` at line 629 whose `<canvas className="absolute inset-0 w-full h-full">` fully covers the root's inner area. Per CSS painting order (backgrounds → box-shadow → content → positioned descendants), a positioned descendant with default `z-auto` paints ABOVE the parent's inset box-shadow. On any cell with media, the canvas completely hides the ring.

**Evidence that the pattern in question CAN render correctly:** `LeafNode.tsx:662` — the file-drag-over highlight — uses `<div className="absolute inset-0 ring-2 ring-[#3b82f6] ring-inset pointer-events-none z-10" data-testid={`drop-target-${id}`} />` as a SIBLING of the canvas-clip-wrapper, with explicit `z-10`. This paints above the canvas and is visible on cells with media. The correct pattern is already in the file — cell-drag just isn't using it.

**The fix:** Mirror the file-drag-over pattern for cell-drag-over. Remove the `isOverThisCell` branch from `ringClass`. Add a new `isOverThisCell &&` overlay div as a sibling of the canvas-clip-wrapper, with `data-testid="drag-over-${id}"` (distinct from file-drop's `drop-target-${id}`).

**What this plan DOES:**
- Restore visibility of the cell-drag accent ring on media cells.
- Add unit test coverage for the overlay's presence/absence and testid.
- Update 28-VERIFICATION.md truth 16 evidence.
- Close the 'no outline on hovered cell' portion of Gap 3 Test 4.

**What this plan does NOT do (explicit scope boundaries — called out in comments and VERIFICATION.md):**

   - Not listed in 28-HUMAN-UAT `missing:` items. The user did not report a selection-outline visibility issue.
   - A codebase-wide audit is more appropriate (other components may use the same pattern).
   - Unifying pan/select/drag-over into one overlay pattern touches selection semantics + pan-mode affordances — medium-risk scope creep for a gap closure.
   - Documented as deferred in 28-VERIFICATION.md with a tracking note.

2. **`DropZoneIndicators` per-zone emphasis (DROP-02 / DROP-03).** The 5 icons still render identically regardless of `activeZone`. Per D-15, this is Phase 29 work. This plan does NOT pull DROP-02/03 forward. The UAT Test 2 complaint "it does not accentuate the icon of the move (swap or extend on specific direction)" is explicitly classified as a **Phase-29 design decision, not a defect** — the 5-icon overlay exists (DROP-01 ✓) and the zone prop is plumbed through; only the per-zone emphasis styling is deferred (D-15).

3. **Unifying the file-drop and cell-drop highlights.** They remain distinct components with distinct testids (`drop-target-${id}` for file-drop, `drag-over-${id}` for cell-drag). They cannot co-occur in practice (a single pointer cannot simultaneously drag-from-OS and drag-a-cell), but preserving distinct testids keeps the semantic boundary explicit for future work.

4. **Ring on empty cells (no media).** Currently works fine — the dashed-border-empty path in `ringClass` remains. The overlay div will also render on empty cells during drag-over, layering above the empty background; visually this is additive, not a regression.
