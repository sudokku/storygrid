<objective>
Fix Gap 2 (from 28-HUMAN-UAT Test 2 — "touch press-and-hold drag + drop") and the 'accelerated ghost' component of Gap 3 (Test 4 — "ghost + zone visuals during drag").

**Two coupled bugs, one root cause tree:**

1. **Accelerated ghost at non-1x canvas scale.** `scaleCompensationModifier` at `src/dnd/adapter/dndkit.ts:116-119` divides dnd-kit's `transform.x/y` by `useEditorStore.getState().canvasScale`. This was designed under the false premise that `<DragOverlay>` renders INSIDE the scaled `canvas-surface` container — if that were true, dividing by scale would cancel the ancestor `transform: scale()` and keep the ghost tracking 1:1. But `<DragOverlay>` uses `createPortal(document.body)` internally (confirmed against `node_modules/@dnd-kit/core/dist/core.esm.js` PositionedOverlay impl in `.planning/debug/zone-visuals-broken.md` and `touch-drag-unreliable.md`) — it renders OUTSIDE any ancestor transform, in pure viewport-space. dnd-kit's transform is already viewport pointer-delta. Dividing it AMPLIFIES movement by 1/scale (~3.3× on mobile at canvasScale ≈ 0.3, 5× at heavy zoom-out). Correct behavior: remove the modifier entirely — no compensation is needed.

2. **Edge zones rarely commit on touch.** dnd-kit runs modifiers BEFORE delivering `delta` to `onDragOver`. `CanvasWrapper.handleDragOver` (lines 86-109) reconstructs the pointer as `{ x: activatorEvent.clientX + delta.x, y: activatorEvent.clientY + delta.y }` — using the AMPLIFIED delta. Both the ghost render and the zone-computation share this over-shifted pointer. Short finger drags land the computed pointer PAST the edge-band boundary into the next cell (or null), so `over` resolves wrong or edge zones get skipped. This is a direct consequence of bug 1. Removing the modifier fixes both in the same stroke.

3. **Zone dead-band on non-square cells (secondary — bundled for convenience).** `computeDropZone` uses a shorter-axis threshold: `Math.max(20, Math.min(w,h) * 0.2)`. `DropZoneIndicators` renders edge bands at 20% of EACH axis independently (`height: 20%` for top/bottom, `width: 20%` for left/right). On a non-square cell (e.g. 100w × 300h) the visible top-band indicator is 60px tall but the compute threshold is 20px — so a pointer at y=40 sits inside the visible arrow but compute returns 'center'. The user aims at a visible zone and gets the wrong outcome. Fix: align compute to the per-axis indicator geometry — it is the visual truth the user targets. THIS IS A SECONDARY FINDING from `.planning/debug/zone-visuals-broken.md`, NOT a required Gap-2 closure — the primary Gap 2 closure happens from modifier removal alone. Task 2 ships this change because it's in the same debug report and a small, targeted fix, but executors/reviewers should understand: even without Task 2, Gap 2 would close.

**Upstream fix — MeasuringStrategy.Always.** @dnd-kit documents (PITFALLS.md:461, issues #50/#205/#250/#393) that `DragOverlay` at non-1x ancestor scale has subtle residual drift unless droppable rects are re-measured every frame via `measuring={{ droppable: { strategy: MeasuringStrategy.Always } }}` on `<DndContext>`. Phase 28 chose the custom modifier instead — wrongly. This plan adopts the upstream recommendation alongside removing the modifier, so there is no drift cause left.

//   keys stay at their defaults (dnd-kit's internal measure fns).
//
// - <DragOverlay> with NO modifiers prop (or empty modifiers={[]}): dnd-kit passes the
//   viewport-space transform through unchanged. This is the target behavior.

// computeDropZone signature is UNCHANGED — only the internal threshold formula changes:
import type { DropZone } from './dragStore';
export function computeDropZone(rect: DOMRect, pointer: {x: number; y: number}): DropZone;

// Target computeDropZone implementation (per-axis threshold aligned with DropZoneIndicators):
export function computeDropZone(rect: DOMRect, pointer: {x: number; y: number}): DropZone {
  const x = pointer.x - rect.left;
  const y = pointer.y - rect.top;
  const w = rect.width;
  const h = rect.height;
  // Per-axis thresholds aligned with DropZoneIndicators.tsx band geometry:
  //   top/bottom bands: height: 20%  → yThreshold = h * 0.2
  //   left/right bands: width: 20%   → xThreshold = w * 0.2
  // 20px floor preserves degenerate-small-cell behavior (CANCEL-05 compatible).
  const yThreshold = Math.max(20, h * 0.2);
  const xThreshold = Math.max(20, w * 0.2);
  // Zone priority (unchanged): top > bottom > left > right > center.
  // Strict-less-than / strict-greater-than preserved so center is closed on its inner edges.
  if (y < yThreshold) return 'top';
  if (y > h - yThreshold) return 'bottom';
  if (x < xThreshold) return 'left';
  if (x > w - xThreshold) return 'right';
  return 'center';
}
