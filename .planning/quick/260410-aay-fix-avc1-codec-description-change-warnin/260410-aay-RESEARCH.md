# Quick Task 260410-aay: Fix avc1 Codec Description Change Warning — Research

**Researched:** 2026-04-10
**Domain:** MediaRecorder codec string, H.264 parameter sets, MP4 fragmented recording
**Confidence:** HIGH

---

## Summary

The browser console warning is generated by Chrome's MediaRecorder implementation when it detects that the H.264 codec description (SPS/PPS parameter sets) changed between frames during a recording session. With `avc1`, the SPS/PPS are stored out-of-band in the MP4 `moov` box header and are expected to remain constant for the entire recording. If they change (e.g., because the encoder reset its internal state), Chrome emits the warning. Switching to `avc3` resolves this: with `avc3`, SPS/PPS are embedded in-band at the start of each fragment, so per-fragment changes are legal by design.

**This is a cosmetic console warning, not a functional error.** The exported MP4 from StoryGrid records from a fixed 1080×1920 canvas, meaning the actual encoding resolution never changes. However, Chrome still emits this warning in some circumstances (see Root Cause below). The fix is a one-line codec string substitution.

**Primary recommendation:** Replace `'video/mp4;codecs=avc1.42E01E'` with `'video/mp4;codecs=avc3.42E01E'` in `src/lib/videoExport.ts` line 274. No architectural changes needed.

## Technical Explanation

### avc1 vs avc3 — Parameter Set Location

| Property | `avc1` | `avc3` |
|----------|--------|--------|
| SPS/PPS location | Out-of-band: stored once in `moov` box header | In-band: prepended to each fragment's first sample |
| Resolution change allowed | No — description must be static for entire file | Yes — each fragment can carry new SPS/PPS |
| Standard | ISO 14496-15 §5.3.2 | ISO 14496-15 §5.3.4 |
| MediaRecorder Chrome support | Chrome 130+ | Chrome 130+ (added alongside avc1) |

[CITED: https://developer.mozilla.org/en-US/docs/Web/Media/Guides/Formats/codecs_parameter]
[CITED: https://www.w3.org/TR/webcodecs-avc-codec-registration/]

### Codec String Format

Both `avc1` and `avc3` use identical syntax after the dot: `[PP][CC][LL]`

- `PP` = profile (hex): `42` = Baseline / Constrained Baseline
- `CC` = constraint flags (hex): `E0` = bits 7,6,5 set (Constrained Baseline Profile flags)
- `LL` = level (hex): `1E` = decimal 30 = Level 3.0

So `avc1.42E01E` = `avc3.42E01E` = H.264 Constrained Baseline Profile Level 3.0.

The codec parameters are identical; only the four-character code prefix changes. [CITED: MDN codecs_parameter]

### Root Cause in StoryGrid

StoryGrid exports from a fixed 1080×1920 canvas. The canvas resolution never changes during recording. However, Chrome's H.264 encoder can reset its internal state (generating new SPS/PPS) when the encoder is first initialized or when the video track starts, and with `avc1` any mismatch between the stored header SPS/PPS and those generated mid-recording triggers the warning.

With `avc3`, Chrome embeds SPS/PPS into each fragment independently, so even if the encoder generates new parameter sets, they are valid in-band and the warning is suppressed. [MEDIUM confidence — inferred from Chrome MediaRecorder README + warning message text]

## The Fix

**Single-line change** in `src/lib/videoExport.ts`:

```typescript
// Before
'video/mp4;codecs=avc1.42E01E',

// After
'video/mp4;codecs=avc3.42E01E',
```

### isTypeSupported check still applies

The existing `MediaRecorder.isTypeSupported()` loop on line 281 will validate the `avc3` string before use, exactly as it does for `avc1`. No change to the detection logic is needed.

### Does avc3 affect playback compatibility?

No. Both `avc1` and `avc3` produce standard H.264 streams in MP4 containers. The codec parameter only tells the muxer where to store the parameter sets. Both are playable in all the same environments. [CITED: ISO 14496-15 / MDN]

## Browser Compatibility

| Browser | avc3 MediaRecorder support |
|---------|---------------------------|
| Chrome 130+ | Supported (added with MP4 MediaRecorder, same release as avc1) |
| Firefox | Not supported for MP4 (uses VP9/WebM — unaffected) |
| Safari | Not supported for MediaRecorder (uses VP9/WebM — unaffected) |

The fallback chain (`video/mp4` → `video/webm;codecs=vp9` → ...) handles Firefox and Safari correctly; `avc3` is only attempted on Chrome 130+. [MEDIUM confidence — no explicit Chromium changelog confirmation found for exact Chrome version]
