  <name>Task 1: Fix overflow split direction alternation in autoFillCells</name>
  <files>src/lib/media.ts, src/test/media.test.ts</files>
  <action>
**NOTE: This fix supersedes decision D-14 from 19-CONTEXT.md.** D-14 specified depth-based direction (`depth % 2`). UAT diagnosis revealed this approach is unreliable: when `splitNode` uses Case B (appending a sibling to an existing parent whose direction already matches), the new node is inserted at the same depth level as the previous split target. The depth does not increase, so `depth % 2` returns the same value on consecutive overflow splits, producing repeated horizontal (or vertical) splits instead of alternating. The corrected approach uses an explicit `overflowCount` counter (even=horizontal, odd=vertical) which reliably alternates regardless of which splitNode case is triggered. D-14 has been updated in 19-CONTEXT.md to reflect this correction.

In `autoFillCells` in src/lib/media.ts, the overflow split direction uses `lastFilledDepth % 2` which fails to alternate when `splitNode` uses Case B (appending sibling to existing parent when parent.direction matches) — depth stays the same so `% 2` returns the same value every iteration.

Fix:
1. Add `let overflowCount = 0;` before the `for` loop (alongside `lastFilledNodeId` and `lastFilledDepth`).
2. In the `else if (lastFilledNodeId !== null)` branch (the overflow split branch, around line 87-94):
   - Replace `const splitDir = lastFilledDepth % 2 === 0 ? 'horizontal' : 'vertical';` with `const splitDir = overflowCount % 2 === 0 ? 'horizontal' : 'vertical';`
   - Add `overflowCount++;` after the `actions.split(lastFilledNodeId, splitDir);` call.
3. In the edge-case branch (line 96-105, single filled root leaf), also use `overflowCount % 2` and increment after split.
4. Remove `lastFilledDepth` variable entirely — it is no longer used. Keep `lastFilledNodeId` since it tracks the split target.
5. Remove `targetDepth` assignments in all three branches since depth is no longer tracked.

Update the test in src/test/media.test.ts — the existing "auto-splits last filled leaf for overflow files (depth-based direction)" test:
- Change description to "auto-splits last filled leaf for overflow files (alternating direction)"
- Add a new test: "overflow splits alternate direction across multiple overflows" that drops 5 files into a single-leaf tree and asserts `actions.split` was called 4 times with alternating directions: horizontal, vertical, horizontal, vertical.

Do NOT change the `splitDir` variable name or the ternary structure — only change the input from `lastFilledDepth` to `overflowCount`.

<task type="auto">
  <name>Task 2: Rewrite detectAudioTrack to use HTMLVideoElement instead of AudioContext</name>
  <files>src/lib/media.ts, src/test/phase19-foundation.test.ts</files>
  <action>
In src/lib/media.ts, rewrite `detectAudioTrack` to use HTMLVideoElement instead of AudioContext.decodeAudioData. The current implementation cannot parse video container formats (MP4, WebM) — AudioContext only decodes raw audio — so it always throws and hits the fail-open catch, returning true for ALL videos.

New implementation of `detectAudioTrack(file: File): Promise&lt;boolean&gt;`:
```
1. Create blob URL: `const url = URL.createObjectURL(file);`
2. Create video element: `const video = document.createElement('video');`
3. Set `video.preload = 'metadata';`
4. Wrap in try/finally where finally always calls `URL.revokeObjectURL(url);`
5. Inside try: return a new Promise that:
   a. Sets video.src = url
   b. Listens for 'loadedmetadata' event
   c. On loadedmetadata:
      - Check `(video as any).audioTracks?.length > 0` (Chrome/Safari — AudioTrackList API)
      - OR check `(video as any).mozHasAudio === true` (Firefox)
      - If either is true, resolve(true)
      - If audioTracks exists and length === 0 AND mozHasAudio is not true, resolve(false)
      - If neither API is available (audioTracks is undefined and mozHasAudio is undefined), resolve(true) — fail-open, browser doesn't support detection
   d. Listens for 'error' event — resolve(true) on error (fail-open)
   e. Add a 5-second timeout that resolves(true) if neither event fires (fail-open)
6. Outer catch: return true (fail-open)
```

Remove the AudioContext import/usage entirely.

Update the JSDoc comment to reflect the new HTMLVideoElement approach. Keep the fail-open behavior documented.

In src/test/phase19-foundation.test.ts, rewrite the `detectAudioTrack` tests to mock HTMLVideoElement instead of AudioContext:

Replace the entire `describe('detectAudioTrack', ...)` block with new tests:

1. Helper: Create a mock video element factory that returns an object with `preload`, `src` setters, `addEventListener` that captures handlers, and configurable `audioTracks` / `mozHasAudio` properties. Use `vi.spyOn(document, 'createElement')` to return the mock when called with 'video'.

2. Mock `URL.createObjectURL` to return `'blob:test'` and `URL.revokeObjectURL` as a no-op.

3. Tests:
   - "returns true when video has audioTracks with length > 0": Set mock `audioTracks = { length: 2 }`. Fire loadedmetadata. Expect true.
   - "returns false when video has audioTracks with length === 0": Set mock `audioTracks = { length: 0 }`. Fire loadedmetadata. Expect false.
   - "returns true when mozHasAudio is true (Firefox)": Set mock `mozHasAudio = true`, no `audioTracks`. Fire loadedmetadata. Expect true.
   - "returns false when mozHasAudio is false and no audioTracks": Set mock `mozHasAudio = false`, no `audioTracks`. Fire loadedmetadata. Expect false.
   - "returns true (fail-open) when neither API is available": No `audioTracks`, no `mozHasAudio`. Fire loadedmetadata. Expect true.
   - "returns true (fail-open) on video error event": Fire error event instead of loadedmetadata. Expect true.
   - "revokes blob URL after detection": Assert `URL.revokeObjectURL` was called with `'blob:test'`.

Remove the old AudioContext-based mocks (`mockClose`, `mockDecode`, `global.AudioContext`).
