<objective>
Fix the 'insert (edge-drop) broken on desktop + touch' portion of 28-UAT Gap 1 (reported twice: Test 1 desktop + Test 2 touch).

**Root cause 1 (desktop + touch) — missing `onDragMove` handler.**

`handleDragOver` at `src/Grid/CanvasWrapper.tsx:88-111` is the ONLY zone-compute site in the app. Per `@dnd-kit/core/dist/core.esm.js`:

- Line 3244-3286: the `onDragOver` callback is dispatched from a `useEffect` whose dependency array is `[overId]`. It fires when `overId` changes — i.e., on droppable ENTER and LEAVE — never on pointer-move within the same droppable.
- Line 3210-3243: the `onDragMove` callback is dispatched from a `useEffect` whose dependency array is `[scrollAdjustedTranslate.x, scrollAdjustedTranslate.y]`. It fires on every pointer-move tick during an active drag.

Result: the zone written to `dragStore.activeZone` is the zone at the moment of first ENTRY into the target cell. Any subsequent pointer motion within the same cell does NOT update the store. When the user releases on an edge band, `handleDragEnd` reads the stale entry-zone — often `'center'` or whatever edge the pointer happened to cross first — and commits the wrong semantic. Users report "insert still not working"; the real bug is "insert always commits the first zone the pointer happened to touch inside the target, not the zone the pointer was in at release."

**Root cause 2 (touch-only amplifier) — unsafe `PointerEvent` cast.**

Line 99-100: `const startX = (activatorEvent as PointerEvent).clientX; const startY = (activatorEvent as PointerEvent).clientY;`

- `CellDragMouseSensor` extends `@dnd-kit/core`'s `MouseSensor`; its activator is bound to the React `onMouseDown` key and receives a DOM `MouseEvent` in `activatorEvent`. `MouseEvent.clientX/Y` are valid — mouse path works.
- `CellDragTouchSensor` extends `@dnd-kit/core`'s `TouchSensor`; its activator is bound to React `onTouchStart` and receives a DOM `TouchEvent` in `activatorEvent`. `TouchEvent.clientX` is `undefined` — top-level `clientX/Y` do not exist on TouchEvent; coordinates live on `touches[0].clientX/Y` (and `changedTouches[0]` for touchend).

Result on touch: `startX = undefined` → `pointer.x = undefined + delta.x = NaN`. `computeDropZone(rect, { x: NaN, y: NaN })` evaluates `y < yThreshold`, `y > h - yThreshold`, `x < xThreshold`, `x > w - xThreshold` — every comparison with NaN is `false`. Falls through to `return 'center'`. On touch, the zone is ALWAYS `'center'` regardless of finger position. Edge-insert is physically unreachable on touch — exactly matching the first-pass UAT report "I can swap them but I can't extend to top/bottom/left/right" which was MISDIAGNOSED in plan 28-12 as the scaleCompensationModifier issue.

**Evidence:** `.planning/debug/insert-edge-drop-broken.md` captures the full debug trace, including the @dnd-kit/core source line numbers cited above and the elimination of competing hypotheses (H1 threshold math — FALSE, aligned; H2 moveCell bug — FALSE, never reached with non-center zone; H3 ghost occlusion — FALSE as causal; H4 DragOverlay collision — FALSE, `over` resolves correctly, only `zone` is wrong).

**The fix:**

1. **Register `onDragMove` on `<DndContext>`.** Move the pointer → rect → computeDropZone → setOver pipeline from `handleDragOver` into a new `handleDragMove`. Retain `handleDragOver` ONLY for null-over / self-over clearing (so `overId` becomes `null` when the pointer leaves the target or re-enters the source cell — this branch of the existing code is correct and must stay).

2. **Replace the unsafe cast with input-type-agnostic pointer derivation.** Derive the pointer from `active.rect.current.initial + delta`:

   ```typescript
   const initial = active.rect.current.initial;  // ClientRect — viewport-space top/left/width/height of the source cell at drag-start
   if (!initial) return;
   // dnd-kit's delta is the cumulative pointer-move vector, normalized across Mouse and Touch.
   // Starting from the center of the source cell and adding delta yields the current pointer
   // position in viewport space, regardless of input type — no activatorEvent access needed.
   const pointer = {
     x: initial.left + initial.width / 2 + delta.x,
     y: initial.top + initial.height / 2 + delta.y,
   };
   ```

   This derivation is correct because at drag-start, the pointer is SOMEWHERE inside the source cell (the sensor's activation constraints guarantee the pointer is on the source when drag begins). Adding the cumulative delta to any stable anchor inside the source — the center is convenient, but any corner would work — yields the viewport-space pointer position. The only requirement is that the anchor matches across all DragMoveEvent callbacks, which `active.rect.current.initial` satisfies (it's the rect at drag-start, frozen for the duration of the drag).

**Alternative derivation (if the executor prefers):**

Branch on `activatorEvent instanceof TouchEvent`:

```typescript
let startX: number, startY: number;
if (activatorEvent instanceof TouchEvent) {
  const t = activatorEvent.touches[0] ?? activatorEvent.changedTouches[0];
  startX = t?.clientX ?? 0;
  startY = t?.clientY ?? 0;
} else if (activatorEvent instanceof MouseEvent) {
  startX = activatorEvent.clientX;
  startY = activatorEvent.clientY;
} else {
  return;
}
const pointer = { x: startX + delta.x, y: startY + delta.y };
```

Either approach closes DEFECT 2. The `active.rect.current.initial` approach is PREFERRED because it removes the type branching entirely and aligns with @dnd-kit's own input-type-agnostic abstractions.

**Target shape after this plan — NEW `handleDragMove` callback:**

```typescript
// Gap-closure 28-14: handleDragMove is the authoritative continuous
// zone-compute pipeline. It fires on every pointer-move tick during an
// active drag (dependency array is
// [scrollAdjustedTranslate.x, scrollAdjustedTranslate.y] — see
// @dnd-kit/core/dist/core.esm.js:3210-3243), which is exactly the semantics
// we need: the store's activeZone must reflect the current pointer position
// at the moment handleDragEnd reads it. Previously, this computation lived
// in handleDragOver which fired only on droppable enter/leave — the zone
// was captured at entry and never refreshed.
//
// Pointer derivation is INPUT-TYPE-AGNOSTIC (gap-closure 28-14 DEFECT 2
// fix): active.rect.current.initial + delta is valid across Mouse and
// Touch input. The previous approach cast activatorEvent to PointerEvent
// and read clientX/Y, which silently yielded `undefined` on TouchEvent
// (top-level clientX/Y do not exist on TouchEvent; they live on touches[0]).
// Undefined + delta = NaN; NaN comparisons in computeDropZone fall through
// to 'center'; touch edge-zone drops were physically unreachable.
const handleDragMove = useCallback((event: DragMoveEvent) => {
  const { over, active, delta } = event;
  if (!over || active.id === over.id) {
    // Continuous clearer — handleDragOver's entry/exit clearer already handles
    // this on droppable-cross, but handleDragMove can hit this path on the
    // first tick before onDragOver fires for the enter event. Keep the
    // store consistent either way.
    useDragStore.getState().setOver(null, null);
    return;
  }

  // Input-type-agnostic pointer: anchor at the center of the source cell at
  // drag-start (active.rect.current.initial is the viewport-space ClientRect
  // of the source at onDragStart, frozen for the drag's lifetime) and add
  // dnd-kit's cumulative delta. delta is already normalized across input
  // types — this derivation works identically for Mouse, Touch, and Pen.
  const initial = active.rect.current.initial;
  if (!initial) return;
  const pointer = {
    x: initial.left + initial.width / 2 + delta.x,
    y: initial.top + initial.height / 2 + delta.y,
  };

  const targetEl = document.querySelector(`[data-testid="leaf-${over.id}"]`) as HTMLElement | null;
  if (!targetEl) {
    useDragStore.getState().setOver(null, null);
    return;
  }
  const rect = targetEl.getBoundingClientRect();
  const zone: DropZone = computeDropZone(rect, pointer);
  useDragStore.getState().setOver(String(over.id), zone);
}, []);
```
