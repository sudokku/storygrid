## Summary

This phase converts StoryGrid from a desktop-only editor into a fully mobile-usable app. The implementation strategy is to add a responsive layer on top of the existing code without rebuilding core components. Three pillars: (1) layout swap at `md` breakpoint using Tailwind responsive prefixes, (2) a new `MobileSheet` component that wraps the existing `CanvasSettingsPanel` and `SelectedCellPanel`, and (3) `CanvasWrapper`'s existing `ResizeObserver + scale` logic handles mobile sizing with minimal changes.

The codebase is already well-structured for this work. All panel content (`CanvasSettingsPanel`, `SelectedCellPanel`) is already split out of `Sidebar`. The `useMediaQuery` hook already exists in `Sidebar.tsx`. `Divider` already uses `setPointerCapture` which fires on touch. The biggest new piece is the bottom sheet gesture/snap mechanism, which is best implemented as a pure CSS+state machine (no external library).

A critical finding: **@dnd-kit's DndContext is not currently wired into `EditorShell` or any component in the rendering tree.** Cell swap uses native HTML5 drag events (dragstart/drop on ActionBar + LeafNode), not dnd-kit's sensor system. D-16's requirement to "ensure TouchSensor is included in DndContext sensor configuration" is moot — the swap mechanism is native drag events, which already work on Android Chrome but **do NOT work on iOS Safari** (iOS does not support HTML5 drag events). This is a known limitation that should be documented. If cell swap on iOS is required, it needs a touch-based alternative.

**Primary recommendation:** Build `MobileSheet` as a new standalone component (not an adaptation of Sidebar) with CSS transform-based sliding, three snap points driven by a `sheetSnapState` field added to `editorStore`. Use `hidden md:flex` / `md:hidden` Tailwind classes as the primary breakpoint mechanism throughout.

### Breakpoint Implementation Pattern

Use Tailwind `md:` prefix (768px) throughout. The existing `useMediaQuery` hook in `Sidebar.tsx` should be extracted to `src/hooks/useMediaQuery.ts` for reuse:

```typescript
// src/hooks/useMediaQuery.ts  (extract from Sidebar.tsx)
export function useMediaQuery(query: string): boolean {
  const [matches, setMatches] = useState(() => window.matchMedia(query).matches);
  useEffect(() => {
    const mql = window.matchMedia(query);
    const handler = (e: MediaQueryListEvent) => setMatches(e.matches);
    mql.addEventListener('change', handler);
    return () => mql.removeEventListener('change', handler);
  }, [query]);
  return matches;
}
```

For CSS-only breakpoints (visibility), prefer Tailwind classes over the hook. The hook is for logic that must respond to breakpoint changes (e.g., "should I render the 3-step onboarding or the welcome card?").

### Canvas Mobile Sizing

The existing `CanvasWrapper` `ResizeObserver` logic:

```typescript
const scaleByH = (height * 0.9) / CANVAS_H;  // 1920
const scaleByW = (width * 0.9) / CANVAS_W;   // 1080
setAutoFitScale(Math.min(scaleByH, scaleByW));
```

On mobile, `CanvasArea` must fill 100vw (no padding). Change `CanvasArea.tsx`:

```tsx
// Desktop: p-8 (32px padding on all sides)
// Mobile: p-0 pt-2 (no side padding)

```

The `ResizeObserver` will then see 100vw width → `scaleByW` becomes the limiting factor on mobile → canvas fills full width. No other changes to `CanvasWrapper` needed (D-13 confirmed).

The `zoom` slider must be bypassed on mobile. On mobile, `zoom` should be locked at 1.0 and the zoom controls hidden. The `canvasScale` will be purely driven by `autoFitScale`. No changes to the scale computation itself — just hide the zoom UI.

### Pan Mode on Mobile (D-11)

Double-tap to enter pan mode: `onDoubleClick` already fires on mobile touch (two quick taps). No changes needed to the entry mechanism.

While in pan mode, the sheet should show a "Done" button. This means `MobileSheet` must read `panModeNodeId` from editorStore and render a "Done" overlay/button at the top of its content area when `panModeNodeId` is set.

Pan within the cell: existing `onPointerDown` / `onPointerMove` / `onPointerUp` handlers already work with touch (touch fires pointer events). No changes needed.

Pinch-to-zoom within a cell: the existing `wheel` handler for zoom does not fire on mobile pinch. Pinch-to-zoom within a cell is not explicitly listed as a locked decision — the existing scroll-to-zoom is desktop-only. D-11 says "scroll/pinch to zoom within the cell" — this requires a `touchstart` + `touchmove` handler reading `e.touches` for pinch distance calculation. This is the most complex new interaction in the phase.

**Pinch detection pattern:**

```typescript
// Add to LeafNode (native touch events, not pointer events):
const handleTouchStart = (e: TouchEvent) => {
  if (e.touches.length === 2) {
    pinchStartDistRef.current = Math.hypot(
      e.touches[1].clientX - e.touches[0].clientX,
      e.touches[1].clientY - e.touches[0].clientY
    );
    pinchStartScaleRef.current = leafState?.panScale ?? 1;
  }
};
const handleTouchMove = (e: TouchEvent) => {
  if (!isPanMode || e.touches.length !== 2) return;
  e.preventDefault();
  const dist = Math.hypot(
    e.touches[1].clientX - e.touches[0].clientX,
    e.touches[1].clientY - e.touches[0].clientY
  );
  const ratio = dist / pinchStartDistRef.current;
  const newScale = Math.max(1, Math.min(3, pinchStartScaleRef.current * ratio));
  updateCell(id, { panScale: newScale });
};
```

The `touchmove` handler must be added with `{ passive: false }` (same as the wheel handler) to call `e.preventDefault()` and prevent page scroll during pinch.

### Anti-Patterns to Avoid

- **Creating a second Sidebar component for mobile:** Don't duplicate `CanvasSettingsPanel` or `SelectedCellPanel` — import them directly into `MobileSheet`.
- **Using `position: fixed` with `bottom: 0` and `height` animation:** Animating `height` causes layout reflow on every frame. Use `transform: translateY()` instead — GPU-composited, 60fps.
- **Relying on `display: none` to hide the desktop Sidebar on mobile:** The sidebar is a flex child; use `hidden md:flex` to keep layout correct. `display: none` will cause layout issues if the parent is a flex container expecting a specific child count.
- **Reading `window.innerWidth` directly instead of `useMediaQuery`:** Already established pattern in codebase — use the hook.
- **Adding `touchstart`/`touchend` instead of `pointerdown`/`pointerup` for the sheet drag:** The Divider already uses pointer events for this reason. Use pointer events for the sheet drag handle too — they work for both mouse and touch.

## Critical Code Findings

### Finding 1: Cell Swap is NOT using @dnd-kit — iOS Drag Events Don't Work

**What the code shows:** Cell swap uses native HTML5 drag events (`draggable`, `onDragStart`, `onDrop`). DndContext from @dnd-kit is never instantiated anywhere in the component tree. There are no `useSensors`, `MouseSensor`, or `TouchSensor` calls.

**Impact of D-16:** D-16 says "ensure TouchSensor is included in DndContext sensor configuration." This is not applicable because DndContext doesn't exist in the tree. The native HTML5 drag API is used instead.

**iOS Safari behavior:** HTML5 drag events (`dragstart`, `dragover`, `drop`) are NOT supported on iOS Safari. Cell swap will silently not work on iOS. On Android Chrome, drag events do work.

**Recommendation for planner:** Document this as a known limitation. If iOS cell swap is needed, it requires a different approach (touch-based pointer events for drag). Since the CONTEXT.md says "no additional work needed" for D-16 but that's based on an assumption that dnd-kit is being used, the planner should note this discrepancy. For MVP mobile, cell swap on iOS being non-functional is acceptable — the feature works via the sheet panel (Select → cell controls in sheet don't include swap, but it's not a core mobile flow).

### Finding 2: Sidebar Has Two Media Queries to Update

`Sidebar.tsx` has `useMediaQuery('(max-width: 1199px)')` and `useMediaQuery('(max-width: 1023px)')`. The `isTooSmall` check (1023px) renders the "works best on desktop" notice. D-02 removes this. The Sidebar component will need the `isTooSmall` branch removed entirely. The `isNarrow` branch (1199px) can remain for desktop-range narrowing but becomes irrelevant if the Sidebar is hidden below 768px.

Cleanest approach: remove both media query branches from Sidebar.tsx since on mobile the Sidebar is replaced by MobileSheet. Sidebar becomes a simple desktop-only component with fixed 280px width.

### Finding 3: CanvasArea Padding Blocks True 100vw Canvas

`CanvasArea.tsx` has `p-8` (32px padding) on the main container. The `ResizeObserver` sees `containerWidth = vw - 64px` (32px each side). On mobile, the canvas would be `vw - 64px` wide, not `100vw`. 

The mobile fix: apply `p-0` on mobile via `p-0 md:p-8`. The `ResizeObserver` in `CanvasWrapper` will then calculate `scaleByW = vw / 1080` and the canvas will fill full width.

The `mt-8` on the inner canvas div (`relative group mt-8`) also adds top margin. On mobile this should be `mt-2 md:mt-8` to give minimal clearance from the toolbar.

### Finding 4: Existing Tests for D-25 Will Break

`phase05-p03-responsive.test.tsx` explicitly tests that `desktop-notice` appears when viewport is below 1024px. D-02 removes this behavior. These tests must be updated or replaced in the plan's Wave 0.

### Finding 5: `useMediaQuery` Hook Should Be Extracted

The `useMediaQuery` function is defined inside `Sidebar.tsx` and is not exported. `Toolbar.tsx`, `Onboarding.tsx`, and the new `MobileSheet` all need it. Extract to `src/hooks/useMediaQuery.ts`.

## Common Pitfalls

### Pitfall 1: Sheet `position: fixed` Fighting CanvasWrapper Transform

**What goes wrong:** `CanvasWrapper` uses `transform: scale(finalScale)` on the canvas surface. Elements with `position: fixed` inside a transformed ancestor are pinned to the transformed context, not the viewport.

**Why it happens:** CSS spec: `transform` creates a new containing block for `fixed` positioned descendants.

**How to avoid:** `MobileSheet` must be rendered as a sibling of the canvas, NOT inside it. In `EditorShell`, `MobileSheet` is added after the `flex flex-1` div containing `CanvasArea` and `Sidebar`. This keeps MobileSheet at the top DOM level, outside any transform context.

**Pattern:**
```tsx
<div className="flex flex-col h-screen w-screen bg-[#0a0a0a]">
  <Toolbar />
  <div className="flex flex-1 overflow-hidden relative">
    <CanvasArea />
    <Sidebar className="hidden md:flex" />
  </div>
  <MobileSheet />   {/* Sibling of canvas area, not child */}
  <Onboarding />
</div>
```

### Pitfall 2: Bottom Sheet Height Consuming Canvas Space

**What goes wrong:** If `MobileSheet` uses `position: fixed`, it overlaps the canvas but the canvas doesn't know to leave space for the sheet. The user sees canvas content behind the collapsed sheet header.

**How to avoid:** Give the mobile layout structure explicit height allocation:
- `EditorShell` is `flex flex-col h-screen`
- `CanvasArea` takes `flex-1` — it fills remaining space above the sheet
- `MobileSheet` is `shrink-0` with a fixed height in collapsed state, and uses absolute/fixed positioning only for the expanded state overlay

Or: use `position: fixed` for the sheet and add a spacer `div` at the bottom of the layout equal to the collapsed height (60px) so the canvas doesn't render behind the sheet handle.

The simplest approach: `MobileSheet` uses `position: fixed; bottom: 0; left: 0; right: 0;` and `EditorShell` adds `pb-[60px] md:pb-0` to the canvas area container to pad out the collapsed sheet height.

### Pitfall 3: Sheet Drag Conflicts with Cell Pointer Events

**What goes wrong:** When the user drags the sheet handle upward, pointer events may fire on underlying canvas cells, selecting them.

**How to avoid:** On `pointerdown` on the sheet drag handle, call `e.stopPropagation()`. Also add `touch-action: none` to the drag handle element to prevent browser scroll from intercepting the touch.

### Pitfall 4: `SelectedCellPanel` Split Actions Missing on Mobile

**What goes wrong:** D-09 requires the sheet to show "Split Horizontal, Split Vertical" buttons for a selected cell. The existing `SelectedCellPanel` in `Sidebar.tsx` does NOT have split buttons — it only has Upload, Clear Media, and Remove Cell. Split actions are currently only in the `ActionBar` (desktop hover overlay).

**How to avoid:** When reusing `SelectedCellPanel` in `MobileSheet`, either:
(a) Add split buttons to the existing `SelectedCellPanel` component (simplest — they'll show on desktop too, which is fine), or
(b) Create a `MobileCellControls` wrapper that adds split buttons above the existing panel.

Option (a) is recommended — adding H/V split buttons to `SelectedCellPanel` is a pure enhancement, not a regression.

### Pitfall 5: Pinch Zoom Conflicts with Sheet Drag

**What goes wrong:** A two-finger gesture on the canvas in pan mode could simultaneously trigger sheet drag.

**How to avoid:** The sheet drag handle is a small specific area (the handle at the top of the sheet). Pan mode is active inside cell bounds. These areas are spatially separate. No conflict expected in practice. If needed: check `e.touches.length > 1` in the sheet drag handler and bail out if multi-touch.

### Pitfall 6: iOS Safari 100vh Bug

**What goes wrong:** On iOS Safari, `100vh` includes the browser chrome (address bar, tab bar). A full-screen sheet using `height: 100vh` will be partially obscured by the browser UI.

**How to avoid:** Use `height: 100dvh` (dynamic viewport height) for the full-screen snap state. `100dvh` is supported in Safari 15.4+. The project targets Safari 15+ so `dvh` may not be universally available. Fallback: use `height: 100%` on the sheet and ensure the parent chain has defined heights to the screen root.

Alternative safe approach: `height: calc(100vh - 60px)` for the "full" snap state, accepting a 60px gap at the top (leaves a visible toolbar). Actually, for full-screen sheet: use the sheet's `transform: translateY(0)` which moves the sheet's natural bottom position to the top of the screen — this doesn't depend on `100vh`.

### Sheet Pointer Drag Handler

```typescript
// Pointer events on drag handle — works for both mouse and touch
const dragStartRef = useRef<{ y: number; snapAtStart: SheetSnapState } | null>(null);

const handlePointerDown = (e: React.PointerEvent) => {
  e.currentTarget.setPointerCapture(e.pointerId);
  dragStartRef.current = { y: e.clientY, snapAtStart: snapState };
  setIsDragging(true);
};

const handlePointerUp = (e: React.PointerEvent) => {
  if (!dragStartRef.current) return;
  const dy = e.clientY - dragStartRef.current.y;
  const { snapAtStart } = dragStartRef.current;
  const THRESHOLD = 50;
  if (dy < -THRESHOLD) {
    // dragged up → expand
    setSnapState(snapAtStart === 'collapsed' ? 'half' : 'full');
  } else if (dy > THRESHOLD) {
    // dragged down → collapse
    setSnapState(snapAtStart === 'full' ? 'half' : 'collapsed');
  }
  dragStartRef.current = null;
  setIsDragging(false);
};
```

### Extracting useMediaQuery

```typescript
// src/hooks/useMediaQuery.ts
import { useState, useEffect } from 'react';

export function useMediaQuery(query: string): boolean {
  const [matches, setMatches] = useState(() => window.matchMedia(query).matches);
  useEffect(() => {
    const mql = window.matchMedia(query);
    const handler = (e: MediaQueryListEvent) => setMatches(e.matches);
    mql.addEventListener('change', handler);
    return () => mql.removeEventListener('change', handler);
  }, [query]);
  return matches;
}
```

## Open Questions

1. **iOS cell swap**
   - What we know: Cell swap uses native HTML5 drag events. iOS Safari does not support these.
   - What's unclear: Is iOS cell swap a hard requirement for this phase or acceptable to leave broken?
   - Recommendation: Document as known limitation. If required, needs touch-based rewrite of the drag-handle → a pointer event approach or dnd-kit with TouchSensor properly wired.

2. **Pinch-to-zoom in pan mode scope**
   - What we know: D-11 mentions "pinch to zoom within the cell." The existing wheel handler is desktop-only.
   - What's unclear: Is pinch zoom inside a cell a must-have for this phase or can it ship without?
   - Recommendation: Implement it (it's a locked decision in D-11). It requires native `touchmove` with `{ passive: false }` on the cell element.

3. **Sheet height on very small phones (320px wide, 568px tall — iPhone SE 1st gen)**
   - What we know: Half-open at 40vh = 227px on 568px screen. Collapsed at 60px. Canvas height = 568 - 48(toolbar) - 60(sheet) = 460px. Canvas scale = 460/1920 = 0.24 → canvas width = 1080 * 0.24 = 259px, which is less than 320px (100vw). So scaleByW (100vw/1080 = 0.30) dominates and canvas fills 320px wide, 569px tall. The collapsed sheet would then overlap the bottom of the canvas.
   - Recommendation: Accept the overlap or reduce collapsed height to 48px on very small screens. Not blocking for MVP.

4. **Export format setting on mobile**
   - What we know: D-08 says format (PNG vs JPEG) is in the sheet panel. The existing `SelectedCellPanel` doesn't have export format controls — these live in `ExportSplitButton`.
   - What's unclear: Exactly where in the sheet panel does export format go? Canvas settings panel or a dedicated export section?
   - Recommendation: Add export format + quality controls to `CanvasSettingsPanel` (or a new "Export" section within the sheet) accessible at all times, not just when a cell is selected.
