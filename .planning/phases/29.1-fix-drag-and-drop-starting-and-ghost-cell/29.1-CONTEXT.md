# Phase 29.1: Fix Drag-and-Drop Starting and Ghost Cell ‚Äî Context

**Gathered:** 2026-04-19
**Status:** Ready for planning

<domain>
## Phase Boundary

Fix two categories of bugs discovered during Phase 29 human UAT:

1. **Ghost cell visual positioning** ‚Äî The ghost image is vertically shifted relative to the pointer grab point when dragging from anywhere other than near the top-left of a cell. Root cause: `grabOffsetModifier` computes ghost dimensions using a uniform aspect-ratio-preserving scale, but the `<img>` in `DragPreviewPortal` applies `max-width: 200px` and `max-height: 200px` independently via CSS, which can distort the rendered ghost and mismatch the modifier's math.

2. **Drag activation** ‚Äî The 8px distance-based `PointerSensor` fires before the 500ms hold completes, allowing accidental drags from quick mouse movements. The 500ms hold-to-drag should be the ONLY activation mechanism.

Out of scope: ghost opacity, snap-back, ESC-cancel, wobble animation, drop-flash, mobile haptics/CSS ‚Äî all resolved in Phase 29 or deferred to Phase 30.

</domain>

<decisions>
## Implementation Decisions

### Ghost ‚Äî Positioning Fix

- **D-01:** The ghost rendering must use a **proportional cap** ‚Äî fit within 200px while strictly preserving aspect ratio. The modifier's ghost dimension calculation (`ghostW`, `ghostH`) must match the actual rendered ghost dimensions in the browser. The current mismatch (CSS caps each axis independently ‚Üí distorted render) is the root cause of the vertical shift.

- **D-02:** Fix approach: ensure the ghost `<img>` renders at the exact same size the modifier computes. Options include:
  - Set only the constraining axis via inline style (e.g., `width: ghostW, height: ghostH` where both are pre-computed with uniform scale) ‚Äî no CSS cap needed
  - OR keep CSS cap but remove the `height` inline override, letting the browser auto-scale the non-constraining axis
  
  The critical invariant: **rendered ghost width √ó rendered ghost height** must equal **ghostW √ó ghostH** from the modifier.

- **D-03:** Ghost opacity stays at 20% (`opacity: 0.2`). No change.

- **D-04:** Ghost size cap stays at 200px. The cap keeps the ghost small and non-obstructive during drag.

### Drop Zone Detection

- **D-05:** Audit the existing `handleDragMove` wiring in `CanvasWrapper.tsx` to verify that `computeDropZone` always uses the pointer's actual viewport position (from dnd-kit's `delta` + `activatorEvent`) and NOT any ghost position or visual offset. The drop zone must be determined entirely by where the pointer is, independent of the ghost visual. Fix only if the audit reveals a bug.

- **D-06:** Drop zone computation invariant: the ghost's visual position or size must never influence which drop zone is active. Users may grab a cell near the bottom ‚Äî the ghost appears shifted ‚Äî but the drop zone should still track the actual pointer correctly.

### Drag Activation

- **D-07:** Remove the distance-based `PointerSensor` (`activationConstraint: { distance: 8 }`) from `CanvasWrapper.tsx`. Keep **only** the delay-based sensor: `activationConstraint: { delay: 500, tolerance: 8 }`. This ensures drag starts only after a deliberate 500ms hold, never from a quick flick or inadvertent mouse movement.

- **D-08:** The `KeyboardSensor` remains (ESC cancel, Phase 29). DND-01 is still satisfied ‚Äî only `PointerSensor` + `KeyboardSensor` are registered.

### Claude's Discretion
- How to validate the `handleDragMove` audit (unit test, visual test, or code read only) ‚Äî Claude's choice.
- Whether to add a regression test covering the 500ms-only activation constraint ‚Äî Claude's choice.

</decisions>

## Specific Observations from UAT

- The vertical shift is proportional to how far down in the cell the grab point is: grabbing at 20% from top ‚Üí ghost accurate; grabbing at 80% ‚Üí ghost is shifted far upward
- No left-right axis shift reported ‚Äî only vertical. (This may be an artifact of typical cell aspect ratios in StoryGrid's portrait layout; the fix must handle both axes correctly.)
- Drop zone accessibility was mentioned as a secondary symptom of the ghost bug. The pointer-based zone computation should already be correct; verify before fixing.
