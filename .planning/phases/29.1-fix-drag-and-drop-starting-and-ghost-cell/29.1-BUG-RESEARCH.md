# Phase 29.1: Bug Research ‚Äî DragOverlay Ghost Position & Collision Detection

**Researched:** 2026-04-19
**Domain:** @dnd-kit/core v6.3.1 ‚Äî DragOverlay modifiers, collision detection, sensor event flow
**Confidence:** HIGH (all claims verified against installed source at `node_modules/@dnd-kit/core/dist/core.cjs.development.js`)

---

## Executive Summary

Both bugs have been traced to their exact source-code line. Bug 1 is a **JSX prop override** ‚Äî `dragListeners.onPointerDown` silently replaces `handlePointerDown` because it is spread last, so `setPointerDown` is never called and the grab-offset modifier guard always fires. Bug 2 is a **dnd-kit architectural constraint** ‚Äî DragOverlay modifiers affect only visual position; collision detection always uses an unmodified dragging rect. The two bugs are independent but their effects compound: Bug 1 makes the ghost snap to the cell's top-left corner, Bug 2 makes the hover zone mismatch the ghost's visual footprint. Both have clean, proven fixes.

## Bug 1 ‚Äî Ghost Always Appears at Cell Top-Left

### Root Cause (VERIFIED)

**Location:** `src/Grid/LeafNode.tsx` lines 601‚Äì604

```tsx
onPointerDown={handlePointerDown}    // ‚Üê registered at position N
{...(!isPanMode ? dragListeners : {})} // ‚Üê spread LAST, overrides onPointerDown
```

In React JSX, props compile to a plain JavaScript object literal. When the same key appears more than once, the last assignment wins. `dragListeners` is the object returned by `useDraggable` ‚Üí `useSyntheticListeners`, and it contains `{ onPointerDown: handler }` (confirmed at `core.cjs.development.js` line 1638, 2358‚Äì2375, 3410).

Spreading `dragListeners` **after** the explicit `onPointerDown` prop means `handlePointerDown` is never registered. The dnd-kit sensor's handler takes its place. `handlePointerDown` ‚Äî which calls `useDragStore.getState().setPointerDown(e.clientX, e.clientY)` ‚Äî is never executed.

**Consequence:** `pointerDownX` and `pointerDownY` in `dragStore` stay at `0` for the entire drag lifetime (they are only reset to `0` by `end()`, never set to non-zero values because `handlePointerDown` does not fire).

**Modifier guard fires:**

```typescript
// grabOffsetModifier in DragPreviewPortal.tsx
if (pointerDownX === 0 && pointerDownY === 0) return transform;
```

`transform` at drag start = `{ x: 0, y: 0, scaleX: 1, scaleY: 1 }` (the initial translate delta before any pointer movement). `PositionedOverlay` has `position: fixed; left: activeNodeRect.left; top: activeNodeRect.top; transform: translate3d(0,0,0)` ‚Äî i.e., exactly at the source cell's bounding rect. The ghost appears at the cell's top-left corner with no offset applied.

**The PITFALL 1 comment in `useCellDraggable.ts` describes the opposite problem** (listeners spread first ‚Üí explicit handler wins ‚Üí drag never fires). The actual code in `LeafNode.tsx` does the opposite (listeners spread last ‚Üí dnd-kit handler wins ‚Üí `handlePointerDown` never fires).

### Evidence Trail

| Step | What Happens | Source |
|------|-------------|--------|
| User presses pointer | PointerSensor's `onPointerDown` fires (from `dragListeners`), NOT `handlePointerDown` | `core.cjs.development.js:1637‚Äì1655`, JSX prop override rule |
| `handlePointerDown` | Never called | JSX object merge semantics |
| `setPointerDown` | Never called | `dragStore.ts:79` |
| `pointerDownX/Y` | Stay `0` | `dragStore.ts:69` (`INITIAL_STATE`) |
| Modifier guard | `if (pointerDownX === 0 && pointerDownY === 0) return transform;` fires | `DragPreviewPortal.tsx:43` |
| Modifier result | `{ x: 0, y: 0, scaleX: 1, scaleY: 1 }` ‚Äî no offset | ‚Äî |
| Ghost position | `position:fixed; left: cell.left; top: cell.top` ‚Äî top-left of cell | `core.cjs.development.js:3658‚Äì3664` |

### Fix for Bug 1

Merge `handlePointerDown` and `dragListeners.onPointerDown` into a single composed handler. Remove the `{...dragListeners}` spread from JSX (or spread only before the pointer handler). The dnd-kit sensor only needs its `onPointerDown` to be called once ‚Äî it can be invoked programmatically inside the composed handler.

**Recommended implementation:**

```tsx
// In LeafNode.tsx ‚Äî replace handlePointerDown + dragListeners spread with:
const handlePointerDown = useCallback((e: React.PointerEvent) => {
  // D-01/D-02: capture grab point BEFORE sensor fires.
  useDragStore.getState().setPointerDown(e.clientX, e.clientY);
  // Call dnd-kit sensor's activator (required for drag to start).
  // dragListeners.onPointerDown is the PointerSensor's activator handler.
  if (!isPanMode) {
    (dragListeners as Record<string, (e: React.PointerEvent) => void>).onPointerDown?.(e);
    return; // pan logic below only for pan mode
  }
  e.preventDefault();
  e.stopPropagation();
  divRef.current?.setPointerCapture?.(e.pointerId);
  const n = findNode(useGridStore.getState().root, id) as LeafNode | null;
  panStartRef.current = { x: e.clientX, y: e.clientY, panX: n?.panX ?? 0, panY: n?.panY ?? 0 };
}, [isPanMode, id, dragListeners]);

// In JSX: remove {...dragListeners} spread entirely ‚Äî the composed handler above
// calls it internally. Spread remaining listeners if any exist (none in v6 for PointerSensor).
// <div onPointerDown={handlePointerDown} ...> (no {...dragListeners})
```

**Alternative (simpler):** Keep the spread but move `onPointerDown` after it:

```tsx
{...(!isPanMode ? dragListeners : {})}  // spread first
onPointerDown={composedHandler}          // explicit handler wins (last assignment)
```

Where `composedHandler` calls BOTH `dragListeners.onPointerDown` and `setPointerDown`. This is the same logic, just different placement.

**Caveat:** `dragListeners` object reference may change between renders. Use `useCallback` with `dragListeners` in the dependency array, or use a ref to capture the latest value.

## Bug 2 ‚Äî Collision Detection Offset Matches Grab Position Offset

### Root Cause (VERIFIED)

**Architecture:** In dnd-kit v6, `DragOverlay` modifiers and `DndContext` modifiers are independent. The `modifiers` prop on `<DragOverlay>` is applied inside the `DragOverlay` component's own `applyModifiers` call (line 3922). It affects only the CSS transform of the overlay element. It does NOT affect the collision rectangle used for drop target detection.

**Collision rect computation** (`core.cjs.development.js` lines 2981‚Äì2987):

```javascript
const collisionRect = draggingNodeRect
  ? getAdjustedRect(draggingNodeRect, modifiedTranslate)
  : null;
```

Where `modifiedTranslate` is computed from DndContext's OWN `applyModifiers` call (line 2956), using DndContext's `modifiers` prop. `CanvasWrapper` passes no `modifiers` to `<DndContext>` ‚Äî only to `<DragOverlay>`. Therefore `modifiedTranslate` = raw `translate` (pointer delta from activation point, no grab-offset correction).

**Result:** `collisionRect` = cell's original screen rect shifted by exactly how far the pointer moved. The ghost is shifted by an additional `centreDelta` (grab-offset correction). These two positions diverge whenever the user grabs the cell somewhere other than its centre.

**Concrete example:**

```
Cell: left=100, top=200, width=200, height=300 (screen space)
User grabs at: (100, 400) ‚Üí bottom-left of cell
pointerDown at T=0: (100, 400)
activationCoordinates = (100, 400)
After drag: pointer at (100, 300) ‚Äî moved UP 100px
translate = { x: 0, y: -100 }

grabOffsetModifier:
  offsetX = 100 - 100 = 0 (left edge)
  offsetY = 400 - 200 = 200 (bottom edge)
  ghostScale = 200/300 = 0.667
  ghostW = 200 * 0.667 = 133px, ghostH = 300 * 0.667 = 200px
  ghostOffsetY = (200/300) * 200 = 133px
  centreDeltaY = 133 - 100 = +33px

Ghost visual position: top: 200 + (-100) + 33 = 133 (shifted UP by 67px net)
collisionRect:        top: 200 + (-100)     = 100 (shifted UP by 100px)

Ghost is visually at top=133, collisionRect is at top=100.
The collision rect is 33px ABOVE the ghost's visual top edge.
```

This matches the reported symptom: dragging from the bottom ‚Üí hovering over the TOP of a droppable (where the ghost appears) does NOT register, because the collisionRect is already past it.

**`rectIntersection` (the default)** uses `collisionRect` as its input ‚Äî it has no access to the pointer position at all (`core.cjs.development.js:429‚Äì462`).

**`pointerWithin`** uses `pointerCoordinates` exclusively (`core.cjs.development.js:479‚Äì519`):

```javascript
const pointerWithin = ({ droppableContainers, droppableRects, pointerCoordinates }) => {
  // Only fires if pointer is literally inside a droppable rect
  if (!pointerCoordinates) return [];
  // checks: isPointWithinRect(pointerCoordinates, rect)
};
```

`pointerCoordinates = activationCoordinates + translate` = current pointer position in viewport space (lines 2913, 2974). This is completely independent of the grab offset or the dragging rect. `pointerWithin` solves Bug 2 because it ignores the misaligned `collisionRect` entirely.

### Fix for Bug 2

Pass `pointerWithin` (with keyboard fallback) as the `collisionDetection` prop on `<DndContext>`:

```tsx
// In CanvasWrapper.tsx ‚Äî add import:
import {
  DndContext, PointerSensor, KeyboardSensor, useSensor, useSensors,
  pointerWithin, rectIntersection   // ‚Üê add these
} from '@dnd-kit/core';

// Composed strategy: pointer-based when available, rect-based for keyboard nav:
const collisionDetectionStrategy = useCallback(
  (args: Parameters<typeof rectIntersection>[0]) => {
    if (args.pointerCoordinates) {
      return pointerWithin(args);
    }
    return rectIntersection(args);
  },
  []
);

// In JSX:
<DndContext
  sensors={sensors}
  collisionDetection={collisionDetectionStrategy}
  onDragStart={handleDragStart}
  onDragMove={handleDragMove}
  onDragEnd={handleDragEnd}
  onDragCancel={handleDragCancel}
>
```

**Why the keyboard fallback is required:** `pointerWithin` returns `[]` when `pointerCoordinates` is null (`core.cjs.development.js:486`). `pointerCoordinates` is null when there is no `activatorEvent` with viewport coordinates ‚Äî this happens with `KeyboardSensor`. Without the fallback, keyboard-initiated drags would never detect any drop target and would always cancel.

**Interaction with `handleDragMove` zone computation:** `handleDragMove` computes the 5-zone drop target using `{ x: ev.clientX + delta.x, y: ev.clientY + delta.y }` as the pointer position (lines 94‚Äì96 of `CanvasWrapper.tsx`). This is the ACTUAL current pointer position (activation pointerdown X + movement delta). This is already correct and is not affected by the collision detection change.

## Are the Two Bugs Related?

Partially. They share a **dnd-kit design constraint** ‚Äî visual position (DragOverlay modifier) and collision detection rect are computed independently ‚Äî but they are triggered by different code paths:

| | Bug 1 | Bug 2 |
|---|---|---|
| Immediate cause | JSX prop override in `LeafNode.tsx` | dnd-kit design: modifiers don't affect collision rect |
| Secondary cause | Guard returns early because `pointerDownX === 0` | Default `rectIntersection` uses dragging rect, not pointer |
| Fix location | `LeafNode.tsx` + `DragPreviewPortal.tsx` (guard may be relaxed) | `CanvasWrapper.tsx` (add `collisionDetection` prop) |
| Fix complexity | Low ‚Äî composed handler | Low ‚Äî add composed strategy |
| Fix independence | Independent of Bug 2 | Independent of Bug 1 |

**Important:** Fixing Bug 1 alone will NOT fix Bug 2. Even with the grab offset correctly applied, the collision detection will still use the misaligned rect. Bug 2 must be fixed separately via `pointerWithin`.

## DragOverlay Positioning Mechanics (for reference)

Understanding how `PositionedOverlay` computes its CSS position explains why the guard in the modifier matters so much on the first frame.

**`PositionedOverlay` styles** (`core.cjs.development.js:3658‚Äì3664`):
```javascript
const styles = {
  position: 'fixed',
  width:  rect.width,   // rect = initialRect = snapshot of activeNodeRect at drag start
  height: rect.height,
  top:    rect.top,     // screen-space top of the draggable at drag start
  left:   rect.left,    // screen-space left of the draggable at drag start
  transform: CSS.Transform.toString(modifiedTransform),
  // ^ this is the only thing the modifier changes
};
```

The overlay's `position:fixed` anchor is always the **draggable node's original screen rect**. The modifier shifts it from there via `transform`. Without the grab offset, `transform = translate3d(0,0,0)` and the overlay sits exactly at the original cell position ‚Äî which visually reads as "appearing at the cell's top-left corner."

**`draggingNodeRect` in the modifier** (`DragOverlay`'s `applyModifiers` at line 3927):
```javascript
draggingNodeRect: dragOverlay.rect
```
`dragOverlay.rect` starts as `null` and is set by a `ResizeObserver` when the `PositionedOverlay` DOM node mounts (line 2445: `setRect(node ? measure(node) : null)`). The measure function is `getClientRect` (getBoundingClientRect, includes transform). On the very first render after drag start, `dragOverlay.rect` is null and the guard fires. After the ResizeObserver fires (typically within the same paint cycle as the first commit), `dragOverlay.rect` is set to the overlay's actual measured rect:

- `.left` = `activeNodeRect.left` (overlay is `position:fixed` at that position, `translate3d(0,0,0)` initially)
- `.width` = `initialRect.width` (the overlay container is sized to the full cell, not to the ghost img)

The ResizeObserver **only updates `.width` and `.height`** on subsequent fires ‚Äî `.left` and `.top` are frozen at the initial mount measurement. CSS transforms on the overlay do not trigger ResizeObserver, so `.left` stays equal to `activeNodeRect.left` throughout the drag. This means the modifier math is geometrically correct once `draggingNodeRect` is non-null ‚Äî the only issue is the `null` guard firing because `pointerDownX/Y = 0`.

## Pitfalls and Caveats

### Pitfall 1: Guard at `pointerDownX === 0`

The guard `if (pointerDownX === 0 && pointerDownY === 0) return transform;` was intended for keyboard-initiated drags. After Bug 1 is fixed and `setPointerDown` is called correctly, a user who physically presses at `clientX=0, clientY=0` (top-left pixel of the viewport, essentially impossible in practice) would still trigger the guard. This is acceptable ‚Äî but if you want to be paranoid, replace the guard with an explicit flag in the drag store instead of relying on zero-coordinate detection.

**Recommended:** Add a `pointerDownSet: boolean` field to `dragStore` and check that instead.

### Pitfall 2: `dragListeners` reference stability

`useDraggable`'s `listeners` object is re-created on every render if `useSyntheticListeners` deps change. When composing `handlePointerDown` with `dragListeners.onPointerDown`, either include `dragListeners` in the `useCallback` dependency array, or capture it in a ref.

```tsx
const dragListenersRef = useRef(dragListeners);
useLayoutEffect(() => { dragListenersRef.current = dragListeners; }, [dragListeners]);
```

### Pitfall 3: `pointerWithin` + fast movement over gaps

`pointerWithin` is a **point-in-rect** test. If the user drags fast, the pointer can skip from one cell to another without passing through the boundary, and both cells will momentarily not detect the pointer. This is not a problem for StoryGrid ‚Äî cells are large (100px+) and `onDragMove` is called for every pointer move event by the PointerSensor.

### Pitfall 4: `useCallback` with stable empty deps for `collisionDetectionStrategy`

The composed strategy function must be stable across renders. Since it only references two imported pure functions (`pointerWithin`, `rectIntersection`), `useCallback(fn, [])` with empty deps is correct.

### Pitfall 5: `sourceW/sourceH` in the modifier are screen-space values

`beginCellDrag` sets `sourceW = rect.width` and `sourceH = rect.height` where `rect = node.getBoundingClientRect()`. These are screen-space dimensions (scaled by `canvasScale`). The modifier uses these to compute `ghostScale`. Since `draggingNodeRect.width` (the overlay's measured width) also equals the screen-space cell width (it's sized to `initialRect.width`), the formula `(offsetX / draggingNodeRect.width) * ghostW` simplifies to `offsetX * ghostScale` ‚Äî dimensionally consistent. This does NOT need to be changed.

## Complete Fix Sketch

### Fix 1: `src/Grid/LeafNode.tsx`

```tsx
// BEFORE (broken ‚Äî dragListeners overrides handlePointerDown):
//   onPointerDown={handlePointerDown}
//   {...(!isPanMode ? dragListeners : {})}

// AFTER ‚Äî composed handler, no spread of dragListeners:
const handlePointerDown = useCallback((e: React.PointerEvent) => {
  // MUST run before anything else ‚Äî captures grab point for grabOffsetModifier.
  useDragStore.getState().setPointerDown(e.clientX, e.clientY);

  if (!isPanMode) {
    // Forward to dnd-kit sensor activator ‚Äî required for drag lifecycle to start.
    (dragListeners as { onPointerDown?: (e: React.PointerEvent) => void }).onPointerDown?.(e);
    return;
  }

  // Pan mode branch (unchanged from previous logic):
  e.preventDefault();
  e.stopPropagation();
  divRef.current?.setPointerCapture?.(e.pointerId);
  const n = findNode(useGridStore.getState().root, id) as LeafNode | null;
  panStartRef.current = {
    x: e.clientX,
    y: e.clientY,
    panX: n?.panX ?? 0,
    panY: n?.panY ?? 0,
  };
}, [isPanMode, id, dragListeners]);  // dragListeners in deps for correctness

// In JSX ‚Äî remove {...dragListeners} spread:
<div
  ref={setRefs}
  {...dragAttributes}
  onPointerDown={handlePointerDown}   // composed handler only ‚Äî no spread
  onPointerMove={handlePointerMove}
  onPointerUp={handlePointerUp}
  // ...rest of props
>
```

### Fix 2: `src/Grid/CanvasWrapper.tsx`

```tsx
// Add to import:
import {
  DndContext, PointerSensor, KeyboardSensor, useSensor, useSensors,
  pointerWithin, rectIntersection,
} from '@dnd-kit/core';
import type { DragStartEvent, DragMoveEvent, DragEndEvent, CollisionDetection } from '@dnd-kit/core';

// Add inside CanvasWrapper component:
const collisionDetectionStrategy = useCallback<CollisionDetection>((args) => {
  // pointerWithin: uses actual pointer position, not the dragging rect.
  // Fixes grab-offset collision mismatch (Bug 2).
  // Falls back to rectIntersection for keyboard-initiated drags where
  // pointerCoordinates is null.
  if (args.pointerCoordinates) {
    return pointerWithin(args);
  }
  return rectIntersection(args);
}, []);

// In JSX:
<DndContext
  sensors={sensors}
  collisionDetection={collisionDetectionStrategy}
  onDragStart={handleDragStart}
  onDragMove={handleDragMove}
  onDragEnd={handleDragEnd}
  onDragCancel={handleDragCancel}
>
```

### Optional: Fix guard in `DragPreviewPortal.tsx`

After Bug 1 is fixed, the guard `if (pointerDownX === 0 && pointerDownY === 0)` is no longer needed for the main flow (it served as an accidental workaround symptom, not a deliberate feature). Keep it as-is for keyboard drag safety, but add a comment clarifying the condition is for keyboard-initiated drags only.
