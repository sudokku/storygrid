<objective>
Fix Gap 1 (BLOCKER from 28-HUMAN-UAT Test 1 + Test 4 desktop): desktop mouse-drag is completely dead because two PointerSensor subclasses register activators under the identical React event key `onPointerDown`. @dnd-kit's `useSyntheticListeners` reducer (`node_modules/@dnd-kit/core/dist/core.esm.js:2361-2376`) writes to a plain object keyed by eventName — the second-registered sensor (Touch) silently overwrites the first (Mouse). The surviving Touch activator rejects `pointerType !== 'touch' && !== 'pen'` → mouse pointerdown events are dropped with no error and no drag lifecycle ever starts on desktop.

**Path chosen: use @dnd-kit's built-in `MouseSensor` (React event key `onMouseDown`) + `TouchSensor` (React event key `onTouchStart`).** These bind to DIFFERENT React event keys and therefore cannot collide under `useSyntheticListeners`. This is the pattern @dnd-kit's own docs recommend for "different constraints for mouse vs touch." D-26 (data-dnd-ignore) is preserved by subclassing each built-in and augmenting the activator with a `closest('[data-dnd-ignore]')` check BEFORE `onActivation`.

**Path rejected: single AbstractPointerSensor subclass with pointerType-switched constraint.** Described in `.planning/debug/desktop-drag-dead.md` as "invasive — must subclass AbstractPointerSensor (not PointerSensor) and feed distinct handleStart timing per pointerType." That path is feasible but carries architectural risk (re-implementing the attach/detach lifecycle) with no benefit over using the two built-ins.

**SC-3 grep gate constraint is materially contradicted by this bug.** The gate's original rationale ("Phase 25 used MouseSensor+TouchSensor combined with 500ms delay and was flaky") is two independent claims: (a) the 500ms delay was wrong (Pitfall 4 — already fixed), and (b) combining the built-ins was flaky. Claim (b) is false — the built-ins coexist correctly because their activators bind to different React event keys. The current PointerSensor-subclass approach is the ACTUAL flaky pattern (proven by this bug). Therefore the SC-3 gate is RELAXED in this gap-closure to permit `MouseSensor` and `TouchSensor` literals. `KeyboardSensor`, `DragZoneRefContext`, and `useDndMonitor` remain forbidden (they encode obsolete Phase 25 architecture; this bug is about MouseSensor/TouchSensor only). A new regression guard BANS the literal `'onPointerDown'` in `src/dnd/adapter/dndkit.ts` — any future revert to the PointerSensor-collision pattern would trip this guard.

D-03 (mouse 8px distance / touch 250ms+5px, never combined) and D-33 (no DOM-drag fallback) are PRESERVED.

// @dnd-kit/core built-in sensors — DIFFERENT React event keys, therefore no listener-key collision
// under useSyntheticListeners (node_modules/@dnd-kit/core/dist/core.esm.js:2361-2376).

// MouseSensor: binds to React 'onMouseDown' — activation governed by distance constraint.
//   Source: node_modules/@dnd-kit/core/dist/sensors/mouse/MouseSensor.d.ts
//   activators = [{ eventName: 'onMouseDown', handler: ... }]
//
// TouchSensor: binds to React 'onTouchStart' — activation governed by delay+tolerance constraint.
//   Source: node_modules/@dnd-kit/core/dist/sensors/touch/TouchSensor.d.ts
//   activators = [{ eventName: 'onTouchStart', handler: ... }]
//
// The handler signature for MouseSensor is different from PointerSensor — it takes a
// React.MouseEvent, not React.PointerEvent. Likewise TouchSensor takes a React.TouchEvent.
// Both built-in handlers accept the same (event, options, activationContext?) shape and
// return boolean. Both call options.onActivation?.({ event }) on success.

// Target class shapes to create (this plan):

import { MouseSensor, TouchSensor } from '@dnd-kit/core';
import type { MouseSensorOptions, TouchSensorOptions } from '@dnd-kit/core';
import type { MouseEvent as ReactMouseEvent, TouchEvent as ReactTouchEvent } from 'react';

export class CellDragMouseSensor extends MouseSensor {
  static activators = [{
    eventName: 'onMouseDown' as const,
    handler: ({ nativeEvent: event }: ReactMouseEvent, { onActivation }: MouseSensorOptions): boolean => {
      // D-26: skip activation if target is inside a [data-dnd-ignore] ancestor.
      const target = event.target as Element | null;
      if (target && typeof target.closest === 'function' && target.closest('[data-dnd-ignore]')) {
        return false;
      }
      // Delegate to parent's activation contract — the built-in handler also checks
      // button === 0 (primary) and event.isPrimary, which we preserve by calling the
      // built-in's original handler via prototype. BUT @dnd-kit's built-in MouseSensor
      // activator handler just calls onActivation({event}) and returns true after the
      // primary-button check — we replicate that here since inheriting static arrays is
      // ES-class-trivial (re-declaration replaces the parent's static).
      if (event.button !== 0) return false;
      onActivation?.({ event });
      return true;
    },
  }];
}

export class CellDragTouchSensor extends TouchSensor {
  static activators = [{
    eventName: 'onTouchStart' as const,
    handler: ({ nativeEvent: event }: ReactTouchEvent, { onActivation }: TouchSensorOptions): boolean => {
      // D-26: skip activation if target is inside a [data-dnd-ignore] ancestor.
      // event.target on a TouchEvent is the element the first touchpoint started on.
      const target = event.target as Element | null;
      if (target && typeof target.closest === 'function' && target.closest('[data-dnd-ignore]')) {
        return false;
      }
      // Mirror @dnd-kit's built-in touch activator: require event.touches.length >= 1.
      if (event.touches.length === 0) return false;
      onActivation?.({ event });
      return true;
    },
  }];
}
