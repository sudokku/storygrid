## Warnings

### WR-01: Stale `pointerDownX/Y` across consecutive drags breaks grab-offset modifier

**File:** `src/dnd/DragPreviewPortal.tsx:43` / `src/dnd/dragStore.ts:76`

**Issue:** `beginCellDrag` resets `overId` and `activeZone` defensively but does **not** reset `pointerDownX`/`pointerDownY`. The guard in `grabOffsetModifier` (`if (pointerDownX === 0 && pointerDownY === 0) return transform;`) only fires when both are exactly zero — i.e. on the very first drag before any `setPointerDown` call. On every subsequent drag session, `pointerDownX/Y` hold the coords from the *previous* drag until `setPointerDown` is called (which happens in `LeafNode.handlePointerDown`, before sensor activation). If `LeafNode.handlePointerDown` is not called — e.g. a drag triggered programmatically, via keyboard, or in a future code path — stale coords propagate silently, shifting the ghost to the wrong position. The guard never triggers because the values are non-zero.

**Fix:** Reset `pointerDownX`/`pointerDownY` to `0` inside `beginCellDrag` in `dragStore.ts`:

```ts
beginCellDrag: (sourceId, ghostUrl, sourceW, sourceH) =>
  set({
    status: 'dragging',
    kind: 'cell',
    sourceId,
    overId: null,
    activeZone: null,
    ghostUrl,
    sourceW,
    sourceH,
    pointerDownX: 0,   // reset so keyboard-drag guard works on every drag
    pointerDownY: 0,
  }),
```

### WR-02: `sourceW`/`sourceH` documented as canvas pixels but supplied as viewport pixels

**File:** `src/Grid/CanvasWrapper.tsx:82` / `src/dnd/dragStore.ts:17`

**Issue:** The `dragStore` JSDoc (lines 16-17) states `sourceW`/`sourceH` are "source cell width in canvas pixels". However, `handleDragStart` populates them from `node.getBoundingClientRect()` (line 82), which returns *viewport* pixel dimensions — already scaled by `finalScale` (e.g. a 540px canvas cell at scale 0.3 reports ~162px). The `grabOffsetModifier` uses these values for ghost-scale computation, and the `CAP = 200` math was written assuming viewport pixels (a viewport-rect cell will rarely exceed 200px at typical zoom levels). The bug is latent: if a developer trusts the "canvas pixels" documentation and derives logic from it, results will be wrong. Additionally, if `finalScale` is very small (e.g. zoom-out), `sourceW`/`sourceH` may be tiny, making `ghostScale` always 1 and the `ghostW`/`ghostH` capping logic effectively dead.

**Fix:** Either correct the dragStore JSDoc comment to say "source cell viewport rect dimensions (scaled)" — or, if canvas pixels are truly intended, divide by `finalScale` before calling `beginCellDrag`:

```ts
// Option A — fix the comment (minimal change, preserves current behavior):
// sourceW : number  (source cell viewport width at current zoom; NOT canvas pixels)
// sourceH : number  (source cell viewport height at current zoom; NOT canvas pixels)

// Option B — supply true canvas pixels:
useDragStore.getState().beginCellDrag(
  sourceId,
  ghostUrl,
  rect.width / finalScale,   // convert back to canvas pixel space
  rect.height / finalScale,
);
```

Option A requires verifying that the `CAP = 200` and modifier logic remain correct with viewport pixels. Option B requires verifying that `CAP = 200` still makes sense for canvas pixels (it would not — a full-canvas cell is 1080×1920px). Option A is the lower-risk fix.

### WR-03: `handleDragStart` early-return does not cancel the dnd-kit drag session

**File:** `src/Grid/CanvasWrapper.tsx:68-70`

**Issue:** When `data-dnd-ignore="true"` is detected, `handleDragStart` returns early without calling `beginCellDrag`, leaving `dragStore.sourceId` as `null`. dnd-kit has already activated the drag at this point (the sensor fired). The drag session proceeds: dnd-kit fires `onDragMove` and `onDragEnd` normally. `handleDragEnd` guards on `!sourceId` and calls `end()` — so no move occurs — but `document.body.style.cursor` is set to `'grabbing'` on line 84 only *after* the early return, so the cursor is not set. However, `handleDragEnd` unconditionally resets the cursor (`document.body.style.cursor = ''`), so if the cursor was `'grabbing'` from a prior drag that wasn't cleaned up, the early-return path cleans it correctly. The real issue: after the early return, dnd-kit still renders `DragOverlay` (its internal `isDragging` is true) — but because `isDragging` in `DragPreviewPortal` reads from `dragStore.status`, the overlay renders `null`. This is safe but wasteful and can cause a blank `DragOverlay` flicker. If dnd-kit exposes a cancel mechanism in a future version this will matter more.

**Fix:** Add cursor cleanup inside the early-return block to make it symmetric with normal end paths:

```ts
if (!node || node.closest('[data-dnd-ignore="true"]')) {
  // dnd-kit session is live but we opt out — ensure cursor is not stuck.
  document.body.style.cursor = '';
  return;
}
```

This does not cancel the dnd-kit session (no public API for that in v6.3.1) but ensures cursor state is always consistent regardless of path.

### IN-01: `window.matchMedia` called on every render without memoization

**File:** `src/dnd/DragPreviewPortal.tsx:79-81`

**Issue:** `prefersReducedMotion` is computed inline on each render by calling `window.matchMedia(...)`. `matchMedia` constructs a new `MediaQueryList` object on every call and is not free. Since `DragPreviewPortal` re-renders during every drag tick (it subscribes to `ghostUrl`, `sourceW`, `sourceH`, `status`), this can be called dozens of times per second during active drags.

**Fix:** Hoist to a module-level constant or use `useMemo`/`useRef`:

```ts
// Module-level (simplest — computed once at import time):
const prefersReducedMotion =
  typeof window !== 'undefined' &&
  window.matchMedia('(prefers-reduced-motion: reduce)').matches;

export function DragPreviewPortal() {
  // ... remove the inline declaration
}
```

### IN-02: `width`/`height` HTML attributes on `<img>` are overridden by inline `style`

**File:** `src/dnd/DragPreviewPortal.tsx:102-103`

**Issue:** The `<img>` element sets `width={sourceW}` and `height={sourceH}` (raw/uncapped values, e.g. 540 viewport pixels) as HTML attributes, but immediately overrides them with `style={{ width: ghostW + 'px', height: ghostH + 'px' }}` (capped values). Inline `style` always wins over HTML attributes for layout. The HTML attributes cause the browser to pre-allocate intrinsic size based on the large raw values before the style is applied, which can cause layout jitter on slower devices. They also mislead readers into thinking the ghost renders at `sourceW×sourceH`.

**Fix:** Remove the `width`/`height` attributes and rely solely on the `style` dimensions:

```tsx
<img
  src={ghostUrl}
  style={{
    opacity: 0.2,
    display: 'block',
    pointerEvents: 'none',
    width: `${ghostW}px`,
    height: `${ghostH}px`,
  }}
  alt=""
  draggable={false}
/>
```

### IN-03: `setTimeout` in `handleDragEnd` is not cancelled on unmount

**File:** `src/Grid/CanvasWrapper.tsx:129`

**Issue:** `setTimeout(() => useDragStore.getState().clearLastDrop(), 700)` fires 700ms after a successful drop. If `CanvasWrapper` unmounts within that window (e.g. route change, app teardown), the callback runs against the Zustand store after unmount. Zustand stores are module-level singletons and handle this gracefully (no error), but the pattern is a known React antipattern that can mask real issues in future refactors.

**Fix:** Store the timeout id in a ref and clear it in a cleanup effect, or accept the current behavior as intentional given Zustand's singleton nature — it causes no visible harm in this codebase.

```ts
const dropFlashTimerRef = useRef<ReturnType<typeof setTimeout> | null>(null);

const handleDragEnd = useCallback(({ over }: DragEndEvent) => {
  // ... existing logic ...
  if (dropFlashTimerRef.current) clearTimeout(dropFlashTimerRef.current);
  dropFlashTimerRef.current = setTimeout(() => {
    useDragStore.getState().clearLastDrop();
  }, 700);
}, [moveCell]);

useEffect(() => () => {
  if (dropFlashTimerRef.current) clearTimeout(dropFlashTimerRef.current);
}, []);
```
