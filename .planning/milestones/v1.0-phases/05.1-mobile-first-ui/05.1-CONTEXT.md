# Phase 05.1: Mobile-First UI - Context

**Gathered:** 2026-04-03
**Status:** Ready for planning

<domain>
## Phase Boundary

Make the StoryGrid editor fully usable on mobile phones. The current Phase 5 decision (D-25) that showed a "works best on desktop" notice below 1024px is **reversed** by this phase. Primary users build collages on their phones — this phase delivers a complete mobile editing experience via a responsive layout swap at the 768px breakpoint.

Scope: responsive layout architecture, bottom sheet panel replacing the desktop sidebar, mobile toolbar condensation, touch-adapted cell controls, wider divider hit targets, and a mobile-specific onboarding card. Export mechanics (Canvas API), store logic, and grid engine are unchanged.

</domain>

<decisions>
## Implementation Decisions

### Responsive Breakpoint
- **D-01:** Mobile layout activates at **< 768px** (Tailwind `md` breakpoint). At ≥ 768px the desktop layout (toolbar + canvas + sidebar side-by-side) is unchanged.
- **D-02:** Phase 5 D-25 ("below 1024px = show a desktop notice") is **removed**. No minimum-width guard on mobile — the app is fully functional at any width.

### Mobile Layout Architecture
- **D-03:** On mobile, the sidebar is replaced by a **bottom sheet panel** that slides up from the bottom of the screen.
- **D-04:** The bottom sheet is **context-sensitive**: default content = Canvas settings panel (gap, border radius, background). When a cell is selected, the sheet content switches to that cell's controls (split H/V, remove, fit toggle, pan/zoom entry). No tabs needed — state drives content.
- **D-05:** The bottom sheet has **three snap states**:
  1. **Collapsed** (~60px): only the drag handle + sheet header row is visible
  2. **Half-open** (~40% viewport height): full controls panel visible for editing
  3. **Full-screen**: entire screen height, for detailed editing when needed
  The user can drag between states or tap the handle to cycle through them.
- **D-06:** The **sheet header row** (visible in collapsed state) contains: Templates | Undo | Redo | Clear icon buttons. This gives one-tap access to the most important non-export actions without opening the sheet.

### Mobile Toolbar
- **D-07:** Mobile toolbar is minimal: **app logo/wordmark on the left, Export button on the right**. All other desktop toolbar controls (undo, redo, zoom, safe zone toggle, templates, clear) are removed from the mobile toolbar — they live in the bottom sheet header (D-06) or sheet panel.
- **D-08:** The Export button on mobile is a single button (not the split button with format popover). Format options (PNG vs JPEG) are accessible via the sheet panel when Export format needs to change.

### Cell Controls on Touch
- **D-09:** Tapping a cell selects it (same as desktop click). When selected, the bottom sheet auto-switches to show that cell's controls: **Split Horizontal, Split Vertical, Toggle Fit (Cover/Contain), Remove Cell, Clear Media** buttons. No floating ActionBar on mobile — the desktop ActionBar is hidden at < 768px.
- **D-10:** The existing desktop ActionBar (hover-based, shows on mouse-over) is **hidden on mobile** via responsive Tailwind class (`hidden md:flex` or equivalent). It remains functional on desktop.
- **D-11:** Double-tapping a selected cell with media enters **pan mode** (same as desktop double-click). While in pan mode, the sheet shows an "Exit pan mode" / Done button. Drag to reposition, scroll/pinch to zoom within the cell. Escape key (if available) or the Done button exits pan mode.

### Canvas Fit on Mobile
- **D-12:** Canvas fills **100% viewport width**. Height scales automatically to maintain 9:16 aspect ratio. The canvas sits above the (collapsed) bottom sheet. No zoom slider on mobile — the canvas always fits the screen width.
- **D-13:** The canvas auto-scales via the existing `CanvasWrapper` CSS transform logic — on mobile, it should compute scale based on 100vw width and fit accordingly.

### Divider Touch Target
- **D-14:** The existing `Divider` component (pointer events + `setPointerCapture`) is **kept as-is** — pointer events fire correctly on mobile touch. The invisible hit area is expanded to **20–24px** wide/tall while keeping the visual line at its current width (4px). This is the only change needed.

### Media Upload on Mobile
- **D-15:** Tapping an empty cell opens the OS file picker (same `input[type=file] accept="image/*"` code path as desktop). On iOS/Android this automatically shows the photo library + camera option. No changes to upload logic needed.

### Cell Swap on Mobile
- **D-16:** @dnd-kit drag handle for cell swap already supports touch sensors (dnd-kit v6). Ensure `TouchSensor` is included in the `DndContext` sensor configuration (it may already be present). No additional work needed.

### Mobile Onboarding
- **D-17:** On mobile (< 768px), the existing 3-step highlight overlay is **suppressed**. Instead, show a **one-time full-screen welcome card** on first load:
  - Content: "Tap cells to select. Tap an empty cell to upload a photo. Hit Export when done."
  - A single "Got it" / Dismiss button.
  - Uses the same `localStorage` key (`storygrid_onboarding_done`) as the desktop overlay — dismissing either one suppresses both.
- **D-18:** The welcome card is a full-screen fixed overlay with a centered card, dark backdrop, and the app logo. Simple, not a multi-step flow.

### Claude's Discretion
- Exact snap animation / spring physics for the bottom sheet (CSS transitions or a library like `react-spring` — keep it lightweight)
- Whether to implement the sheet as a new component or adapt the existing `Sidebar` with responsive display logic
- Exact pixel heights for the three sheet snap states (collapsed ~60px, half-open ~40vh, full-screen ~100vh are targets)
- Touch gesture threshold for sheet drag (how many px of drag before snapping to next state)
- Whether zoom controls (zoom %, +/− buttons) appear anywhere on mobile at all, or are completely removed
- Safe zone toggle on mobile — can be in the sheet panel or omitted entirely

</decisions>

<specifics>
## Specific Ideas

- Bottom sheet feel: like Figma mobile or Canva mobile — familiar to the target user
- Sheet header icons (Templates | Undo | Redo | Clear) should be visible in collapsed state — this is the key discoverability mechanism for actions on mobile
- Canvas should fill full screen width; no letterboxing or padding on the sides on mobile
- Welcome card on first load is a full-screen overlay (not a small toast) — needs to feel intentional

</specifics>
